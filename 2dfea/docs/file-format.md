# 2dfea Model File Format — v1.0.0

This page describes the JSON file format used by 2dfea's **Export JSON**
and **Import JSON** features (toolbar buttons + `Ctrl+S` / `Ctrl+O`).

The file format is also the canonical shape for the `localStorage`
autosave entry (`2dfea-model-storage`). One serializer, two sinks.

## Overview

Every exported file has the top-level shape:

```jsonc
{
  "schemaVersion": "1.0.0",
  "metadata": {
    "appVersion": "unknown",
    "exportedAt": "2026-04-27T15:35:00.000Z",
    "name": "Cantilever sample",         // optional
    "description": "Single 4 m beam",    // optional
    "author": null,                      // v1 always null
    "units": { ... }                     // pinned literal — see below
  },
  "model": {
    "nodes":            [ /* unit-suffixed entities */ ],
    "elements":         [ /* unit-suffixed entities */ ],
    "loads":            { "nodal": [...], "distributed": [...], "elementPoint": [...] },
    "loadCases":        [ ... ],
    "loadCombinations": [ ... ],
    "activeLoadCase":   "Dead",
    "activeResultView": { "type": "case", "name": "Dead" },
    "idCounters":       { ... }
  },
  "_comments": {
    "nodes":    { "N1": "Fixed support" },
    "elements": { "E1": "Generated by AI: IPE 200 in S275" },
    "loads":    { "NL1": "Tip dead load" }
    // loadCases, loadCombinations also supported
  }
}
```

The full machine-readable contract is the published JSON Schema:

- File: [`public/schemas/2dfea-model-v1.json`](../public/schemas/2dfea-model-v1.json)
- URL (post-deploy): https://magnusfjeldolsen.github.io/structural_tools/2dfea/schemas/2dfea-model-v1.json

The schema is **regenerated from the Zod source** by the `prebuild` script,
so the published JSON Schema and the runtime validator never drift.

## Unit policy (v1, locked)

Numeric fields carry their unit in the field name. v1 is unit-fixed:

| Field suffix | Meaning              |
|--------------|----------------------|
| `_m`         | metres (length)      |
| `_kN`        | kilonewtons (force)  |
| `_kNm`       | kilonewton-metres (moment) |
| `_kN_per_m`  | kilonewtons / metre (distributed load intensity) |
| `_GPa`       | gigapascals (stress / Young's modulus) |
| `_m4`        | m⁴ (area moment of inertia) |
| `_m2`        | m² (area)            |
| `_C`         | degrees Celsius (future thermal loads) |

`metadata.units` is a literal object pinned to:

```json
{ "length": "m", "force": "kN", "moment": "kNm",
  "stress": "GPa", "areaMomentOfInertia": "m4", "area": "m2" }
```

Future versions MAY allow user-selected units; v1 does not.

## Versioning policy (SemVer)

`schemaVersion` is a **SemVer string**, not an integer. Bump rules:

- **MAJOR** (`1.x.x → 2.0.0`): a breaking change requiring migration
  (renames, restructures, removed fields). Old code rejects newer-MAJOR
  files with a clear toast.
- **MINOR** (`1.0.x → 1.1.0`): an additive, optional change (e.g.
  `Element.sectionRef`, `loads.thermal`, `metadata.tags`). Old code
  parses the file via `.passthrough()`, emits a single deduplicated
  `console.warn` listing every unknown key, and ignores them.
- **PATCH** (`1.0.0 → 1.0.1`): doc-only or non-breaking enum **expansion**
  (e.g. a new `support` type that older code wouldn't see in older files).

The forward-compat policy is enforced by `.passthrough()` on the
`metadata`, `model`, `model.loads`, `model.idCounters`, and every entity
object. Only the top-level envelope (`schemaVersion`, `metadata`, `model`,
`_comments`) is strict.

## Key design rules

1. **Unit-suffixed field names** — every numeric field with a physical
   unit names that unit (`x_m`, not `x`). Self-documenting; AI-friendly.
2. **Stable insertion order in arrays**, sorted keys in objects. The
   exporter is deterministic — round-tripping a file produces a
   byte-identical second export (modulo `metadata.exportedAt`).
3. **No `null`-vs-`undefined` ambiguity** — optional fields are
   omitted when absent. Exception: `metadata.author` is always emitted
   as `null` in v1 (placeholder for a future field).
4. **`idCounters` is part of `model`**, not metadata — counters are
   model-authored data and round-trip-relevant.
5. **`_comments` is sibling to `model`**, not nested into entities. The
   leading underscore signals "annotation, not part of the analysis
   model". Comments are stored at runtime in a parallel
   `comments: Record<string, string>` map so `types.ts` entity shapes
   stay stable.
6. **Direction enums are case-sensitive**:
   - Distributed load `direction`: `Fx | Fy | FX | FY`
     (lowercase = local, uppercase = global).
   - Element-point load `direction`: same plus `Mz | MZ`.

## AI-prompt suitability

The schema is designed so an AI structured-output prompt can attach the
published JSON Schema as a hard constraint, and the LLM provider's
structured-output mode emits conforming JSON in one shot. See
[plan §5.15](plans/save-load-json.md#515-ai-prompt-suitability) for the
grounded-generation flow.

The `_comments` block lets the AI (or a human author) annotate **why**
each entity was created — useful for downstream review and for chained
prompts that need to read the rationale.

## Round-trip guarantee

For any v1 file `A`:

```text
canonicalStringify(modelStateToFile(applyToStore(parse(A)))) === A
```

…after masking `metadata.exportedAt` (the timestamp updates on each
export). This contract is enforced by the test suite at
`src/io/roundtrip.test.ts`.

Unknown fields from a future MINOR file are **not** preserved through
a v1 round-trip — they are accepted on import (and `console.warn`'d) but
dropped on re-export, since v1's runtime store has nowhere to put them.
A v1.1.0 reader would preserve them.
